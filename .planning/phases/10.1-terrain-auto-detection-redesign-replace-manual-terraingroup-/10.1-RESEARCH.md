# Phase 10.1: Terrain Auto-Detection Redesign — Research

**Researched:** 2026-05-01
**Domain:** Godot 4.6 TileSet TerrainSet auto-detection + dual-grid terrain boundary dispatch simplification
**Confidence:** HIGH

## Summary

Phase 10.1 replaces the Phase 10 `PentaTileTerrainGroup` Resource + per-terrain layout array with auto-detection: terrain count is derived from `atlas_grid_size.y` across all `TileSetAtlasSource` sources, Godot native `TerrainSets` store names/colors (storage-only, never used for solving), and a single shared `layout: PentaTileLayout` drives all terrains. Spike 009 validated all 6 auto-detection behaviors headless against Godot 4.6.2.

The approach is significantly simpler than the Phase 10 implementation: ~400 LOC deleted (TerrainGroup Resource, terrain_index Dictionary, per-corner dual-grid terrain dispatch, per-terrain Penta synthesis bridge), ~120 LOC added (auto_detect_terrains, simplified _resolve_terrain_id, snapshot-before-clear name/color persistence, tile_set.changed signal wiring, per-corner same-terrain sample_fn filtering).

All variation machinery (SINGLE/PROBABILITY/STRIP modes, deterministic hash, probability weights), source_id routing, terrain_mode() virtual, compute_mask(strip_index) signature, and PentaTileLayoutSlope survive unchanged. The refactor is a clean simplification — the Phase 10 terrain dispatch behavior is preserved through a different (simpler) mechanism.

**Primary recommendation:** Implement `_auto_detect_terrains()` as a single function of ~50 LOC following the spike's proven pattern (compute terrain count from atlas grid, snapshot old names/colors, clear+recreate terrain_set 0, assign TileData.terrain, restore user-set names/colors). Wire it to `tile_set` setter + `tile_set.changed` signal via the existing deferred coalescer pattern. Simplify dual-grid dispatch to per-corner same-terrain filtering via a terrain-aware sample_fn passed to compute_mask.

## Implementation Details

### 1. _auto_detect_terrains() — Core Implementation (~50 LOC)

This is the heart of the refactor. Based on spike 009's validated `auto_setup_terrains()`, adapted for `PentaTileMapLayer` as a private method:

```gdscript
## Auto-detect terrain count from atlas grid dimensions across all sources.
## Creates terrain_set 0, assigns TileData.terrain = sequential atlas row ID.
## Idempotent — preserves user-set terrain names/colors via snapshot-before-clear.
## Called from tile_set setter and on tile_set.changed (deferred coalescer).
func _auto_detect_terrains() -> void:
	if tile_set == null:
		return

	# 1. Compute terrain count: sum atlas_grid_size.y across all TileSetAtlasSource sources
	var infos: Array = []  # [{source_id: int, first_terrain_id: int, row_count: int}]
	var next_id: int = 0
	for src_idx in range(tile_set.get_source_count()):
		var sid: int = tile_set.get_source_id(src_idx)
		var src := tile_set.get_source(sid) as TileSetAtlasSource
		if src == null:
			continue
		var rows: int = src.get_atlas_grid_size().y
		if rows <= 0:
			continue
		infos.append({"source_id": sid, "first_terrain_id": next_id, "row_count": rows})
		next_id += rows

	var total: int = next_id
	if total == 0:
		return  # No atlas sources with rows — nothing to do

	# 2. Snapshot old names/colors BEFORE clearing (D-14: user-set customizations survive)
	var old_names: Dictionary = {}
	var old_colors: Dictionary = {}
	if tile_set.get_terrain_sets_count() > 0:
		for i in tile_set.get_terrains_count(0):
			old_names[i] = tile_set.get_terrain_name(0, i)
			old_colors[i] = tile_set.get_terrain_color(0, i)

	# 3. Clear + recreate terrain_set 0
	while tile_set.get_terrain_sets_count() > 0:
		tile_set.remove_terrain_set(0)

	tile_set.add_terrain_set()

	# 4. Select terrain mode from active layout's terrain_mode() (D-07)
	# Default MATCH_CORNERS if layout is null or terrain_mode() returns -1
	var mode := TileSet.TERRAIN_MODE_MATCH_CORNERS
	var resolved_layout := _resolve_layout()
	if resolved_layout != null:
		var layout_mode := resolved_layout.terrain_mode()
		if layout_mode >= 0:
			mode = layout_mode
	tile_set.set_terrain_set_mode(0, mode)

	# 5. Add terrains + restore names/colors (D-14)
	for i in range(total):
		tile_set.add_terrain(0)
		# Name: restore user-set, regenerate auto-names
		var name: String = "Terrain %d" % i
		if old_names.has(i):
			var old := str(old_names[i])
			if not old.begins_with("Terrain "):
				name = old  # preserve user-set name
		tile_set.set_terrain_name(0, i, name)
		# Color: restore user-set, regenerate auto-colors via deterministic hash
		if old_colors.has(i):
			tile_set.set_terrain_color(0, i, old_colors[i])
		else:
			tile_set.set_terrain_color(0, i,
				Color.from_hsv(float(hash("tc%d" % i) % 256) / 255.0, 0.7, 0.8))

	# 6. Assign TileData.terrain = sequential terrain ID (D-12, D-15)
	for info in infos:
		var sid: int = int(info.source_id)
		var src := tile_set.get_source(sid) as TileSetAtlasSource
		if src == null:
			continue
		var first: int = int(info.first_terrain_id)
		var rows: int = int(info.row_count)
		for row in range(rows):
			var tid: int = first + row
			for col in range(src.get_atlas_grid_size().x):
				var coord := Vector2i(col, row)
				if src.has_tile(coord):
					for alt in range(src.get_alternative_tiles_count(coord)):
						var td := src.get_tile_data(coord, alt)
						if td:
							td.terrain_set = 0
							td.terrain = tid

	# 7. Trigger full rebuild
	_queue_rebuild()
```

**Key design decisions:**
- **Uses `_resolve_layout()` not `self.layout`** — matches existing pattern; null layout returns 0 terrains (no-op)
- **terrain_mode() from resolved layout** — covers all 9 subclasses correctly; MATCH_CORNERS default is safe for unknown/abstract layouts
- **Sequential terrain IDs span all sources** — source 0 rows 0..R0-1 → IDs 0..R0-1, source 1 rows 0..R1-1 → IDs R0..R0+R1-1
- **All alternative tiles get terrain assigned** — per spike 009, iterates `get_alternative_tiles_count(coord)`
- **No peering bits set** — PentaTile's own solver generates visuals; peering bits would be misleading if set incorrectly

### 2. tile_set.changed Signal Wiring (~10 LOC)

Follow the existing `layout.changed` pattern (disconnect-before-reconnect, deferred coalescer):

```gdscript
# In tile_set setter (extend existing logic):
# tile_set setter currently handles _tile_set_is_fallback tracking.
# ADD after tile_set is written:
	_auto_detect_terrains()

# NEW signal handler:
## Receives TileSet.changed when the bound TileSet is externally mutated
## (add_source, remove_source, etc.). Coalesces via _queue_rebuild's
## call_deferred — multiple emissions per frame collapse to one rebuild.
func _on_tile_set_changed() -> void:
	_auto_detect_terrains()
```

**Connection management:**
- Connect in `tile_set` setter (after writing `tile_set` value)
- Disconnect old connection before connecting new (prevent double-fire on reassign)
- Disconnect in `_exit_tree()` or when tile_set is set to null (prevent dangling signal)

**Pattern (inside tile_set `_set` hook or setter override):**
```gdscript
# In _set(property, value) — extend the existing tile_set hook:
func _set(property: StringName, value: Variant) -> bool:
	if property == "tile_set" and not _suppress_tile_set_override:
		if value != tile_set:
			# Disconnect old tile_set.changed
			if tile_set != null and tile_set.changed.is_connected(_on_tile_set_changed):
				tile_set.changed.disconnect(_on_tile_set_changed)
			_tile_set_is_fallback = false
		# Connect new tile_set.changed
		if value != null and value is TileSet:
			if not value.changed.is_connected(_on_tile_set_changed):
				value.changed.connect(_on_tile_set_changed)
			# Auto-detect terrains AFTER tile_set is written
			# (deferred so Godot completes the property write first)
			_auto_detect_terrains.call_deferred()
	return false
```

**Caution:** The `tile_set` property's `_set` hook returns `false` to let Godot apply default property behavior. Auto-detection must run AFTER Godot writes the property, so use `call_deferred()` to defer until the next frame when `tile_set` is guaranteed set.

### 3. Simplified _resolve_terrain_id() (~15 LOC)

Read directly from `TileData.terrain` — no TerrainGroup, no custom data layer, no atlas_coords.y encoding:

```gdscript
## Resolve which terrain_id a logic cell belongs to.
## Reads TileData.terrain directly from the cell's source tile data.
## Returns 0 when the cell is empty or tile_set is null.
func _resolve_terrain_id(cell: Vector2i) -> int:
	if tile_set == null:
		return 0
	var source_id := get_cell_source_id(cell)
	if source_id == -1:
		return 0  # empty cell
	var source := tile_set.get_source(source_id) as TileSetAtlasSource
	if source == null:
		return 0
	var atlas_coords := get_cell_atlas_coords(cell)
	if not source.has_tile(atlas_coords):
		return 0
	var tile_data := source.get_tile_data(atlas_coords, 0)
	if tile_data == null:
		return 0
	if tile_data.terrain < 0:
		return 0
	return tile_data.terrain
```

**Key simplifications:**
- **No custom data layer check** — `penta_terrain_id` custom data layer was Phase 10's override mechanism; deleted with TerrainGroup
- **No atlas_coords.y encoding** — terrain identity is in TileData.terrain, not coords.y
- **No terrain_group null guard** — auto-detection runs on every tile_set change, so TileData.terrain is always populated
- **Returns 0 for empty/unresolved** — safe default; layout dispatch still works (terrain 0 tiles render for unknown cells)

### 4. Simplified Dual-Grid Dispatch (~40 LOC)

Replace `_paint_dual_grid_terrain()` (71 lines) with inline per-corner dispatch in `_paint_via_layout()` (~20 lines):

```gdscript
# In _paint_via_layout, dual-grid terrain dispatch section:
# Replace lines 378-384 (old terrain_group guard + _paint_dual_grid_terrain call)
# with:

if active_layout.is_dual_grid():
	_primary_layer.erase_cell(display_cell)

	# Per-corner dispatch: each corner reads terrain_id from its logic cell
	# and dispatches through the SHARED layout's mask_to_atlas.
	# sample_fn filters: only same-terrain neighbors count as filled.
	const CORNERS := [
		Vector2i(-1, -1),  # TL
		Vector2i(0, -1),   # TR
		Vector2i(-1, 0),   # BL
		Vector2i(0, 0),    # BR
	]

	var corner_terrain_ids: Array[int] = []  # Per-corner terrain IDs for variant sample_fn
	for i in range(CORNERS.size()):
		var logic_cell: Vector2i = display_cell + CORNERS[i]
		corner_terrain_ids.append(_resolve_terrain_id(logic_cell) if _has_logic_cell(logic_cell) else -1)

	# Build per-corner mask: each corner is "filled" if its logic cell is painted
	# AND the logic cell is same-terrain as the corner being evaluated.
	# For each unique terrain present in the 4 corners, paint that corner's tile.
	# Each corner renders exactly once — through the terrain of its own logic cell.
	var painted_terrains: Dictionary = {}  # terrain_id -> mask_bits
	for i in range(CORNERS.size()):
		var tid: int = corner_terrain_ids[i]
		if tid < 0:
			continue
		if not painted_terrains.has(tid):
			painted_terrains[tid] = 0
		painted_terrains[tid] |= (1 << i)

	# For each terrain, compute which corners of THAT terrain are filled
	# (only same-terrain corners count as filled). Dispatch through shared layout.
	for tid: int in painted_terrains.keys():
		var terrain_mask: int = painted_terrains[tid]
		if terrain_mask == 0:
			continue
		# Build terrain-aware sample_fn: only cells painted with THIS terrain count as filled
		var terrain_sample_fn := func(logic_cell: Vector2i) -> bool:
			if not _has_logic_cell(logic_cell):
				return false
			return _resolve_terrain_id(logic_cell) == tid

		# Recompute full mask using terrain-aware sample_fn (D-05, D-10)
		var full_mask: int = active_layout.compute_mask(display_cell, terrain_sample_fn, tid)

		if full_mask == 0:
			continue

		var slot := active_layout.mask_to_atlas(full_mask, tid)
		if slot == null:
			continue

		var paint_source: int = source
		if slot.source_id >= 0:
			paint_source = slot.source_id

		_paint_with_slot(_primary_layer, slot, display_cell, paint_source)

	return  # dual-grid path done — skip single-grid + variation below
```

**Key design decisions:**
- **Each corner renders once** — per-corner terrain_mask built from TileData.terrain; no precedence stacking (D-06)
- **sample_fn filters by terrain** — `compute_mask` with a Callable that only returns true for same-terrain cells; other-terrain cells count as empty, creating natural boundaries
- **No per-corner per-terrain layout dispatch** — single shared layout, strip_index = terrain_id
- **No terrain_precedence** — deleted with TerrainGroup; each corner's tile is the final visual for that corner

### 5. Simplified Single-Grid Terrain Dispatch (~10 LOC)

# SINGLE-GRID PATH (after dual-grid check + passthrough check):
if not active_layout.is_dual_grid() and not sample_fn.call(display_cell):

_primary_layer.erase_cell(display_cell)

### 6. _pick_variation_tile() — Simplified (no _terrain_index)

Without `_terrain_index`, variation candidates are discovered at render time by scanning the atlas row for tiles matching the terrain:

```gdscript
func _pick_variation_tile(display_cell: Vector2i, _layout: PentaTileLayout, _mask: int, terrain_id: int) -> Vector2i:
	var source_id: int = _resolve_source_id()
	if source_id < 0 or tile_set == null:
		return Vector2i(-1, -1)
	var source: TileSetAtlasSource = tile_set.get_source(source_id) as TileSetAtlasSource
	if source == null:
		return Vector2i(-1, -1)

	# Scan atlas for tiles sharing the same terrain_id.
	# terrain_id = atlas_coords.y (single-row-per-terrain convention).
	var matching: Array[Vector2i] = []
	var weights: Array[float] = []

	var grid_size: Vector2i = source.get_atlas_grid_size()
	for col in range(grid_size.x):
		var coord := Vector2i(col, terrain_id)
		if not source.has_tile(coord):
			continue
		var tile_data: TileData = source.get_tile_data(coord, 0)
		if tile_data == null:
			continue
		if tile_data.terrain != terrain_id:
			continue
		var prob: float = tile_data.probability
		if prob <= 0.0:
			prob = 1.0
		matching.append(coord)
		weights.append(prob)

	if matching.is_empty():
		return Vector2i(-1, -1)

	# Deterministic hash (D-09, PITFALLS §2)
	var seed_value: int = hash(Vector4i(display_cell.x, display_cell.y, terrain_id, variation_seed))
	var rng := RandomNumberGenerator.new()
	rng.seed = seed_value
	var pick: int = rng.rand_weighted(weights)
	return matching[pick]
```

## Common Pitfalls

### Pitfall 1: tile_set.changed Signal Storm During Auto-Detection
**What goes wrong:** `_auto_detect_terrains()` calls `tile_set.add_terrain_set()`, `add_terrain()`, and `td.terrain = ...` — all of which fire `tile_set.changed`. If `_auto_detect_terrains` is connected to `tile_set.changed` directly, this creates infinite recursion.
**Why it happens:** Godot's `TileSet.changed` fires on ANY mutation to the TileSet, including the mutations made by auto-detection itself.
**How to avoid:** Use a guard flag: `if _performing_auto_detect: return` at the top of `_on_tile_set_changed`. Set `_performing_auto_detect = true` before calling `_auto_detect_terrains()` and `false` after. ALTERNATIVE: disconnect the signal before auto-detection, reconnect after (cleaner, no persistent flag).
**Recommended pattern:**
```gdscript
func _on_tile_set_changed() -> void:
	# Guard against self-triggering: disconnect while auto-detecting,
	# then reconnect after. Prevents infinite recursion when auto-detection
	# mutates the TileSet (add_terrain, set_terrain_name, td.terrain = ...).
	if tile_set != null and tile_set.changed.is_connected(_on_tile_set_changed):
		tile_set.changed.disconnect(_on_tile_set_changed)
	_auto_detect_terrains()
	if tile_set != null and not tile_set.changed.is_connected(_on_tile_set_changed):
		tile_set.changed.connect(_on_tile_set_changed)
```
**Warning signs:** Godot hangs or crashes with stack overflow; test never completes.

### Pitfall 2: Deferred Auto-Detection Races Against _ready()
**What goes wrong:** `_auto_detect_terrains.call_deferred()` in `_set` hook defers the call. If `_ready()` fires between the `_set` hook and the deferred call, the first rebuild runs before terrains exist, producing empty visuals.
**Why it happens:** Deferred calls execute after the current frame's processing. `_ready()` is called during scene tree entry, which may be the same frame as property deserialization.
**How to avoid:** The `tile_set` setter in `_set` defers `_auto_detect_terrains`, but `rebuild()` is called in `_ready()` via `rebuild.call_deferred()`. Since both use `call_deferred`, Godot's deferred call queue is FIFO — the auto-detection runs BEFORE the rebuild (the deferred auto-detect was queued first during property deserialization). **This is already correct** — D-04 confirms the setter fires during scene deserialization, covering the initial-load case.
**Warning signs:** First paint after scene load shows empty cells; subsequent paints work.

### Pitfall 3: snapshot-before-clear Loses Colors When terrain_count Changes
**What goes wrong:** User has 3 terrains, customizes colors, then adds a 4th atlas row. Snapshot only saves colors for IDs 0..2. After rebuild, terrain 3 gets auto-color. Then user removes row 2 (terrain IDs shift — was 0,1,2,3 → now 0,1,3). Old color for terrain 2 is lost; terrain_id 3's old color is applied to terrain_id 2.
**Why it happens:** Terrain IDs are sequential and positional. Removing a row shifts IDs.
**How to avoid:** Accept this as a known limitation. The snapshot preserves names/colors by terrain_id index. When atlas rows are added/removed, terrain IDs shift. This matches Godot's native behavior (removing rows from a spritesheet shifts indices). Users can always re-color via the inspector.
**Warning signs:** Terrain colors shift unexpectedly after resizing atlas; user reports "my floor turned blue."

### Pitfall 4: Variation Picker Performance Without _terrain_index
**What goes wrong:** `_pick_variation_tile` scans the entire atlas row every call instead of using a cached index.
**Why it happens:** The `_terrain_index` Dictionary is deleted (D-01). Without it, variation candidates are discovered at render time.
**How to avoid:** Accept the scan. Demo-scale (~100-1k cells) means at most 1k scans of ~5 columns each = 5k iterations. Regeneration during rebuild (determinism) requires identical scan — no shimmer. For Phase 10.1, the performance is acceptable. The `atlas_grid_size.x` is small (< 16 typically) and scan is O(cols). **If profiling shows this is a hotspot, re-add a lightweight cache (Dictionary[int, Array[Vector2i]] mapping terrain_id → candidate list) rebuilt during auto-detection.**
**Warning signs:** Rebuild takes > 100ms on demo-scale; visible lag on large maps.

### Pitfall 5: `_set` Hook tile_set Connection Race
**What goes wrong:** `_set` hook disconnects old `tile_set.changed` but the new tile_set value hasn't been written by Godot yet (the `_set` hook returns `false`, then Godot applies the default property write). If auto-detection runs before the property write completes, it reads the old tile_set.
**Why it happens:** The `_set` hook is an interceptor — the actual property write happens AFTER the hook returns.
**How to avoid:** Use `call_deferred()` in the `_set` hook for auto-detection. By the time the deferred call executes, Godot has written the `tile_set` property. The deferred call reads `self.tile_set` (which is now the new value). The signal connection is made directly (not deferred — disconnect old, connect new happens synchronously in the hook, before the deferred auto-detect).
**Warning signs:** Auto-detection creates terrains from the OLD tile_set after tile_set reassignment.

## Open Questions

1. **Should `_pick_variation_tile` get a lightweight cache?**
   - What we know: Without `_terrain_index`, variation scans atlas row per call (O(cols)). Demo-scale is fine.
   - What's unclear: Whether real-world usage with large atlases (PixelLab 8×8, 64 tiles per terrain) will cause noticeable lag during rebuild.
   - Recommendation: Ship without cache first. Add a lightweight `Dictionary[int, Array[Vector2i]]` cache built in `_auto_detect_terrains()` if profiling shows a need. The cache maps terrain_id → candidate coords for that terrain.

2. **Should `compute_mask` be called per-terrain for dual-grid, or should each corner dispatch independently?**
   - What we know: D-05 says per-corner single-shared-layout dispatch with sample_fn filtering. The current design calls `compute_mask` once per unique terrain present in the 4 corners (max 4 calls per display cell).
   - What's unclear: Whether calling `compute_mask` once per terrain is always correct for all 9 layout subclasses. Wang2Edge/Min3x3 use edge-based masks (different sample_fn semantics).
   - Recommendation: The dual-grid layouts (Penta, DualGrid16, Wang2Corner, PixelLab) all use corner masks — `compute_mask` with terrain-aware sample_fn produces correct corners. Wang2Edge/Min3x3 are single-grid only. Blob47Godot is single-grid. No dual-grid layout uses edge masks. This design is correct for all current layouts.

3. **What happens to `terrain_mode()` on `PentaTileLayoutSingleTile`?**
   - What we know: `PentaTileLayoutSingleTile` was added in Phase 2.1. It likely has a `terrain_mode()` override (or inherits base's -1).
   - What's unclear: Whether SingleTile needs a terrain set mode or should inherit MATCH_CORNERS default.
   - Recommendation: Check `PentaTileLayoutSingleTile.terrain_mode()` return value. If -1 (base), the MATCH_CORNERS default in `_auto_detect_terrains` is correct for a single-tile layout. Single-tile slicing produces 5 archetypes that map to corner masks.

4. **Should `_auto_detect_terrains` suppress `tile_set.changed` during its own mutations?**
   - What we know: Pitfall 1 describes the self-triggering problem. The recommended fix is disconnect-before-reconnect in `_on_tile_set_changed`.
   - What's unclear: Whether `tile_set.changed` is emitted synchronously during `add_terrain_set()` / `add_terrain()` calls or deferred.
   - Recommendation: Implement disconnect-before-reconnect as described in Pitfall 1. This is safe regardless of whether emission is synchronous or deferred. The spike 009 code did not handle this (it was a standalone function, not connected to a signal) — but the signal wasn't wired in the spike.
