## Key decisions during execution

- **Hybrid orchestration via HANDOFF.md across two Claude sessions.** Plans 01.1-01 + 01.1-02 ran inline in one Claude session. Plan 01.1-03's user-side gates (GitHub rename, local-dir rename) and Plan 03 Tasks 5–6 ran across two sessions — the original session set up the rename and wrote `HANDOFF.md`, and a fresh session at the new path closed out the wrap-up commit. This kept user-side actions discoverable and resumable without holding open an idle Claude conversation.
- **Memory-dir migration was a no-op by the time we checked.** The encoded directory was already at the new name (`c--Programming-Files-Shilocity-PentaTile`) when verification ran, presumably because the Windows-case-insensitive parent local-dir rename made the encoded-name rename effectively atomic. No manual `mv` needed; verification confirmed `MEMORY.md` + the two memory entry files are present.
- **Godot binary path discovery.** The path `CLAUDE.md` documents (`C:\Programming_Files\Godot\Godot_v4.6.2-stable_win64.exe`) is actually a *directory* containing the real `.exe`. The nested binary lives at `...\Godot_v4.6.2-stable_win64.exe\Godot_v4.6.2-stable_win64.exe`. The console wrapper at `...\Godot_v4.6.2-stable_win64_console.exe` is the most reliable choice for headless verification on Windows because it captures stdout/stderr correctly. `HANDOFF.md` documents this for the next session; updating CLAUDE.md is deferred (Action 6 in HANDOFF).
- **Plan 01.1-01 atomic-commit deviation noted.** The first executor agent committed only the `git mv` renames (where git's rename detection saw 0-byte content changes) and left content edits uncommitted; orchestrator caught this and landed a follow-up commit immediately. No half-renamed working tree window. Documented in `01.1-01-SUMMARY.md`.
- **Acceptable leftover references.** Two categories of "leftover" `Tetra` references are intentional and preserved: (1) `CHANGELOG.md` references `TetraTile` because the entire purpose of that entry is to document the rename; (2) `TETRA1` / `TETRA4` / `TETRA5` enum-member names in `02-CONTEXT.md` / `02-DISCUSSION-LOG.md` / `STATE.md` are GDScript enum members for the planned Phase 2 `TileCountMode` enum (numerical prefix referring to "1/4/5 tiles per strip", not the project name). The Phase 1.1 token map deliberately did not include `Tetra[0-9]` patterns. Phase 2 planning will revisit whether those enum members should be renamed (e.g., `STRIP_1`/`STRIP_4`/`STRIP_5`) when the GDScript code is written.
