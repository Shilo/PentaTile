## Class Structure

| Option | Description | Selected |
|--------|-------------|----------|
| Two separate subclasses | `PentaTileLayoutPixelLabTopDown` + `PentaTileLayoutPixelLabSideScroller` — cleaner inspector picker, self-contained, idiomatic GDScript 2 | ✓ |
| One combined class with `mode` enum | `PentaTileLayoutPixelLab` with `{TOP_DOWN, SIDE_SCROLLER}` enum — single file, smaller surface | |

**Auto-selection rationale:** REQUIREMENTS.md PIXLAB-01/02 explicitly name two separate subclasses; spike 003 §"Architecture for TetraTile" recommends two for cleaner inspector picker; consistent with the rest of the layout library convention (one subclass per atlas convention). The Penta layout is the only mode-enum exception because it has load-time auto-detection — PixelLab variants share no detection logic, so there's no upside to a combined class.

## Variation-Bank Semantics (v0.2)

| Option | Description | Selected |
|--------|-------------|----------|
| First-occurrence row-major pick (deferred bank pick to v2) | `mask_to_atlas` returns the first cell in row-major scan order for each mask. Caches `_first_cell_by_mask: Array[Vector2i]` once at Resource load. Bank pick lands when VAR-PIXEL-01 reopens. | ✓ |
| Ship deterministic-hash bank pick now | Add `variation_seed: int` to layouts; `mask_to_atlas(mask, coord)` does `hash(coord, variation_seed) % cells.size()`. Requires base-class signature change. | |
| Random pick at paint time | `randi()` per call — unstable across `rebuild()`; rejected by Pitfall #2. | |

**Auto-selection rationale:** REQUIREMENTS.md PIXLAB-03 explicitly defers bank pick to v2 backlog as VAR-PIXEL-01, design-coupled with VAR-01 and MULTITERR-01 (all three need joint Y-axis decision-making before any of them ship). Shipping bank pick alone would force an asymmetric API change that gets undone when variation work reopens. Per CLAUDE.md no-forward-compat rule, no `variation_seed` field is exposed in v0.2 — when bank pick lands, the API breaks cleanly with a CHANGELOG entry.

## No-Forward-Compat Enforcement

| Option | Description | Selected |
|--------|-------------|----------|
| Reject `pixellab_version: int` field | Per CLAUDE.md HARD RULE — no version markers, no forward-compat scaffolding. If PixelLab updates tables, we CHANGELOG and update constants. | ✓ |
| Add `pixellab_version: int = 1` field as recommended in spike 003 + PIXELLAB.md | Insurance against future PixelLab plugin breaking changes | |

**Auto-selection rationale:** Both spike 003 and PIXELLAB.md were written before the no-forward-compat policy was codified (2026-04-26). Project policy now overrides their recommendation. The version field has no consumer and would only "matter" if PixelLab ships a new plugin AND we update PentaTile to handle both old and new tables — which violates no-forward-compat in the cleanest possible way. CHANGELOG is the only acceptable compat surface.

## Constants & Caching

| Option | Description | Selected |
|--------|-------------|----------|
| Per-subclass duplication of `_CELL_TO_ROLE` / `_ROLE_TO_MASK` / `_MASK_TO_ROLE` | Self-contained subclasses; idiomatic GDScript 2 (class-level `const` cannot reference cross-class consts at parse time) | ✓ |
| Shared base helper file | One `pixellab_constants.gd` with the role-to-mask tables; subclasses import | |

**Auto-selection rationale:** GDScript 2 parse-time const limitations bit the Penta layout in Phase 2 (per CLAUDE.md "Coding Conventions" / `_SLOT_*` constants comment). Same workaround applies. Constants are 16 ints (role-to-mask) + 64 ints (cell-to-role) — small enough that duplication doesn't burden maintenance. Each subclass can be deleted independently.

## Atlas Grid & Bundled PNGs

| Option | Description | Selected |
|--------|-------------|----------|
| 8×8 grid, 32px tiles, greybox-generated PNGs via `_generate_bitmasks.py` | Matches Phase 2 generator + LAYOUT-07 co-located convention | ✓ |
| 8×8 grid, 16px tiles | Matches PixelLab's `reference_image_size = 16` directly | |
| Ship real PixelLab samples as bundled fallback | "Authentic" greybox; commits commercial AI art into the addon | |

**Auto-selection rationale:** 32px matches every other shipped layout; consistency wins. PixelLab samples are commercial AI output — committing them into `addons/` is awkward (licensing implications even if the user owns the samples; not appropriate for an addon distributed via GitHub Release zip). Greybox + cell-to-role-respecting silhouettes give clean prototyping art.

## Visual Regression Test Strategy

| Option | Description | Selected |
|--------|-------------|----------|
| Three-layer test: `pixellab_first_cell_test` + matrix extension + `pixellab_visual_regression_test` | Locks first-cell pick (D-89), extends 6→8 layouts in `comprehensive_bitmask_test`, validates against real PixelLab sample structurally | ✓ |
| Single comprehensive test only | Just extend the matrix; rely on existing test infrastructure | |
| Pixel-perfect PNG-equality vs PixelLab sample | Direct bytes match | |

**Auto-selection rationale:** Three layers track Phase 2/3 methodology (CLAUDE.md Test Methodology). Pixel-perfect equality fails because PentaTile reads variation cells PixelLab's exporter discards — direct PNG comparison wouldn't hold even when the layout is correct. Structural assertions (bbox + non-empty + first-cell deterministic) survive both this constraint AND future bank-pick changes.

## File Layout

| Option | Description | Selected |
|--------|-------------|----------|
| Flat siblings under `addons/penta_tile/layouts/` | Two `.gd` files + two `.png` files at root, matching DualGrid16/Wang2*/Min3x3/Blob47Godot pattern | ✓ |
| Subfolder per layout (like Penta's `penta_tile_layout_penta/`) | One subfolder per PixelLab variant | |

**Auto-selection rationale:** LAYOUT-07 explicitly: "Single-PNG layouts" (DualGrid16, Wang2Edge, Wang2Corner, Min3x3, Blob47Godot, **future PixelLab + TBT**) use flat siblings. Subfolder is reserved for multi-variant layouts (Penta has 10 PNGs across 5 modes × 2 axes).
