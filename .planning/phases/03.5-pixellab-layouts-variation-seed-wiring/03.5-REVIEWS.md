# Cross-AI Plan Review — Phase 3.5

## Codex Review

## Summary

The Phase 3.5 plan set is generally strong: it decomposes the work cleanly into assets, two layout subclasses, contract tests, visual regression, and closeout traceability. It is tightly coupled to the locked decisions D-88..D-105, correctly avoids `variation_seed` and `pixellab_version`, and appears sufficient to achieve PIXLAB-01..04. The main weaknesses are around test brittleness and a few places where the plans over-specify implementation by pasting full files, while also under-specifying some actual semantic checks in the visual regression test.

## Strengths

- Clear dependency ordering: generator/assets first, layout classes second, first-cell tests third, visual regression fourth, documentation closeout last.
- Correctly preserves the v0.2 boundary: first-cell deterministic pick only, no `variation_seed`, no forward-compat `pixellab_version`.
- Good separation between top-down and side-scroller subclasses per D-88, with independent files and explicit copy-paste guardrails.
- PIXLAB-03 is well covered by `pixellab_first_cell_test`: 16 masks × 2 layouts, explicit mask=0 assertions, transform invariants.
- The first-cell cache approach is pragmatic and low-risk: O(64) init cost, simple dispatch, no coordinate-aware API leakage.
- The plans explicitly protect the single-grid `mask=0` trap, which is likely the most important semantic edge case.
- Traceability closeout is unusually thorough: requirements, roadmap, state, deferred VAR-PIXEL-01 preservation, and LOC tracking are all addressed.
- No meaningful security surface is introduced.

## Concerns

- **HIGH**: Plan 05's visual regression test claims to assert "every painted display cell renders to a non-transparent atlas region," but the provided test body does not actually check each blitted tile for opacity before or after composition. It only checks that the final canvas has at least one opaque pixel and that canvas dimensions match. A layout could dispatch several cells to transparent mask=0 tiles and still pass.

- **HIGH**: The Plan 05 verify-the-regression cycle is weak. Forcing `mask_to_atlas` to always return `Vector2i(0, 0)` may not fail the automated assertions if that atlas cell is non-empty and deterministic. The plan even says the output will "look obviously broken" and asks to eyeball the saved PNG. That is not a reliable regression test.

- **MEDIUM**: The "4×4 region exercising all 16 mask states" language is inconsistent with the actual test pattern note. The plan later admits full 16-mask coverage is not load-bearing because Plan 04 covers it. That is fine, but the success criteria overstate what Plan 05 proves.

- **MEDIUM**: The visual regression test does not compare against an expected composed canvas, expected atlas coords, expected per-cell opacity, or expected mask distribution. It is more of a smoke/integration test than a true visual regression.

- **MEDIUM**: Several plans instruct full-file verbatim creation. That reduces ambiguity, but it can also bypass adapting to actual repository patterns discovered during implementation. It is especially risky for GDScript syntax details such as inline one-line `if` statements and typed loop variables if the local code style/compiler expectations differ.

- **MEDIUM**: LOC guardrail is partially respected for runtime code but knowingly exceeded for tests. That is probably acceptable, but Plan 05 normalizes the overage instead of tightening the test to higher signal. The identity guardrail risk remains real because runtime LOC had already exceeded the informational trigger before this phase.

- **LOW**: Duplicating `_CELL_TO_ROLE` and `_ROLE_TO_MASK` across generator, two layout classes, and tests creates drift risk. The first-cell and bounds tests catch some drift, but future edits will require touching many copies.

- **LOW**: Plans rely on `.gd.uid` and `.png.import` generation by headless Godot. That is consistent with the project, but the exact Godot executable path and shell heredoc examples are brittle on Windows PowerShell.

- **LOW**: Plan 06's closeout asks for actual commit hashes from an already-executed phase. Good for traceability, but if commits were squashed or amended, the plan format can become noisy or stale.

## Suggestions

- Future visual regression tests should assert per-cell renderability explicitly: for each painted visual cell, extract the dispatched atlas tile and require at least one opaque pixel unless that specific expected mask is allowed to be transparent.

- Replace the Plan 05 "eyeball saved PNG" regression check with an automated failure condition. For example, assert the set of atlas coords used contains more than one coordinate, or assert expected mask-to-atlas coords for a known pattern.

- Separate "visual smoke test" from "visual regression test" terminology. Plan 05 mostly proves fixture loading, fallback TileSet creation, dispatch bounds, and deterministic rebuilds. It does not strongly prove visual equivalence.

- Future plans should include a compact expected mask distribution for composed-canvas tests. Example: paint a pattern with known expected masks per cell, then assert `primary.get_cell_atlas_coords(cell)` matches `layout.mask_to_atlas(expected_mask)`.

- Keep the first-cell contract test style. It is the strongest test in the phase and would have caught most table, mapping, and cache regressions early.

- Consider adding a tiny shared test helper or generated table verifier in future phases to reduce repeated role/mask constants across generator, layout, and tests. Do this only in tests or tooling, not runtime, to avoid growing the public surface.

- For future GDScript plans, prefer "create file matching this structure" plus key snippets over full verbatim files unless exact content is essential. This leaves room for implementation to follow existing repo idioms.

- Continue preserving deferred requirements explicitly in closeout. The VAR-PIXEL-01 handling is a good precedent for avoiding accidental scope creep.

## Risk Assessment

**Overall risk: MEDIUM.**

The shipped runtime design appears low-risk and directly satisfies PIXLAB-01, PIXLAB-02, and PIXLAB-03. The main residual risk is test confidence around PIXLAB-04: the composed-canvas test is useful, but weaker than its wording suggests, and its verify-the-regression cycle depends partly on human inspection. Scope control and no-forward-compat adherence are strong. Identity/LOC risk remains medium because the broader v0.2 layout library has already grown past the informational LOC trigger, even though this phase's runtime contribution is modest.

   test, not a visual regression test.

   (LOW). First-cell test catches some classes of drift; a tooling-only table
