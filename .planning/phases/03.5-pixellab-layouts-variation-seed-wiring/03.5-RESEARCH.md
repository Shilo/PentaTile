â€” D-89/D-90/D-91 defer the bank pick to v2 (`VAR-PIXEL-01`), design-coupled with

### Pattern 1: Single-grid corner-mask layout subclass

**What:** Subclass `PentaTileLayout`, override the 5 virtuals: `is_dual_grid` (false), `compute_mask` (4-bit corner), `mask_to_atlas` (cached lookup), `_default_bitmask_template_path` (bundled PNG path), `_fallback_atlas_grid_size` (8Ă—8).

**When to use:** Any single-grid 4-bit-corner layout. Same shape as Wang2Corner; PIXLAB adds a per-mask cell cache instead of a formula.

### Pattern 2: First-cell pick via row-major iteration

**What:** Iterate `(row, col)` in row-major order; for each cell compute its mask via the bijection; only set `_first_cell_by_mask[mask]` if the slot is still `Vector2i(-1, -1)` (unfilled sentinel). Mirrors PixelLab's own `tileset_transform.lua:73` ("Only store first occurrence").

**When to use:** Single-grid layouts where multiple atlas cells share the same mask and you need a deterministic pick without exposing per-coord randomness.

**Why row-major matters:** For PIXLAB top-down, role 6 (mask 15) appears 28 times. The first occurrence is `(0, 0)` (top-left of the 8Ă—8 atlas). The bank pick (v2) will use the full set of 28 cells; v0.2 ships only `(0, 0)`. This is acceptable per D-89 â€” matches PixelLab's own exporter behavior.

### Pattern 3: Bundled greybox PNG generator extension

**What:** Add two functions to `_generate_bitmasks.py`: `gen_pixel_lab_top_down()` and `gen_pixel_lab_side_scroller()`, each returning a 256Ă—256 PIL Image. Each iterates the cell-to-role table and draws a per-role silhouette at each cell's atlas position.

**When to use:** Any layout with NĂ—N atlas where cells with the same role need visually-identical greyboxes.

**Example (projected â€” based on existing `gen_blob_47_godot` and `gen_wang_2_corner`):**

### Anti-Patterns to Avoid

- **Putting `_ROLE_TO_MASK` on the base class.** Violates D-98. GDScript 2 cross-class const limitation + pollutes the abstract surface.
- **Adding `variation_seed: int` to layouts.** Violates D-90. Defer to v2 with the rest of the variation work.
- **Adding `pixellab_version: int` to layouts.** Violates D-92. CHANGELOG documents future PixelLab changes.
- **Special-casing `mask == 0` in `mask_to_atlas`.** Unnecessary â€” D-104 / Pitfall #9: the cached `_first_cell_by_mask[0]` returns a valid cell (top-down (2,2), side-scroller (0,0)) by construction. Adding an `if mask == 0: return null` would BREAK isolated-cell rendering.
- **Iterating in column-major order in `_init_cache`.** First-cell pick depends on row-major scan (matches PixelLab's own exporter `tile_map[tile_num] == nil then -- Only store first occurrence`). Column-major would give different first cells for some masks.
- **Caching at parse time via const.** GDScript 2 const initializer cannot run iteration loops; computing `_first_cell_by_mask` at `_init` is the idiomatic pattern.
- **Sharing `_init_cache` between subclasses via the base class.** Each subclass has its own `_CELL_TO_ROLE`; the base class can't see it (per-subclass const, D-98). Inline `_init_cache` in each subclass.
- **Using `_first_cell_by_mask` as a `const Array`.** Const arrays are fine for read-only data, but `Array[Vector2i]` populated at runtime is a `var`, not a const. The two are not interchangeable in GDScript 2.

## Common Pitfalls

### Pitfall 1: GDScript 2 cross-class const limitation

**What goes wrong:** Trying to declare `const _ROLE_TO_MASK := PentaTileLayoutPixelLabTopDown._ROLE_TO_MASK` on the side-scroller class.

**Why it happens:** GDScript 2 class-level `const` initializers cannot reference another class's `const` at parse time â€” the type-resolution pass that registers `class_name` runs after const initialization. CLAUDE.md "Coding Conventions" + STATE.md note this same constraint hit Phase 2's Penta layout `_SLOT_*` consts.

**How to avoid:** Per D-98, duplicate `_ROLE_TO_MASK` in each subclass. 16 ints Ă— 2 subclasses = 32 lines of duplicated declaration; not a maintenance burden since the table is locked by spike 003 and PIXLAB's upstream API.

**Warning signs:** "Identifier not found" parse errors at class load. "Compile error: cannot reference X.Y in const context."

### Pitfall 2: First-cell pick mismatched between top-down and side-scroller

**What goes wrong:** Using PIXLAB top-down's `_first_cell_by_mask` for the side-scroller (or vice versa) â€” the cells are different even though the bijection is identical.

**Why it happens:** Both subclasses share `_ROLE_TO_MASK` but each has its own `_CELL_TO_ROLE`. Top-down mask 0 = (2, 2); side-scroller mask 0 = (0, 0) (D-104). Top-down mask 15 = (0, 0); side-scroller mask 15 = (4, 2).

**How to avoid:** Build `_first_cell_by_mask` in each subclass's own `_init_cache` from its own `_CELL_TO_ROLE`. The cache is var (per-instance), not const (shared). `pixellab_first_cell_test.gd` MUST verify both subclasses independently with hardcoded expected coords.

**Warning signs:** Visual regression shows top-down content rendering for a layout marked side-scroller. `pixellab_first_cell_test` fails for one subclass but not the other.

### Pitfall 3: Mask=0 incorrectly treated as erase

**What goes wrong:** Adding `if mask == 0: return null` in `mask_to_atlas`. Isolated cells (1Ă—1 paints, single corner of a sparse pattern) render nothing.

**Why it happens:** Pitfall #9 is widely repeated â€” single-grid `mask=0` is NOT erase. The layer's universal `mask == 0 â†’ return` short-circuit (`_paint_via_layout` line 281) ONLY applies to dual-grid. Single-grid logic-painted cells with mask=0 (no painted neighbors) MUST dispatch to a default atlas slot. CONTEXT.md D-104 anchors this for PIXLAB.

**How to avoid:** Don't special-case mask=0. The cached `_first_cell_by_mask[0]` returns role 12's first cell â€” top-down (2, 2), side-scroller (0, 0). Both are valid atlas coords; both render. The `pixellab_first_cell_test.gd` MUST include an explicit assertion for mask=0 (analogous to `blob_47_collapse_test.gd:46-50`).

**Warning signs:** Single 1Ă—1 painted cells render empty. `comprehensive_bitmask_test` fails on the `1x1` and `3_isolated` patterns for both PIXLAB layouts.

### Pitfall 4: Variation-seed wiring leak from a draft commit

**What goes wrong:** A wave-2 plan adds `variation_seed: int` to `PentaTileLayout` base "for future use" because the phase title still says "+ Variation-Seed Wiring."

**Why it happens:** ROADMAP.md Phase 3.5 entry "Wire `variation_seed` deterministic-hash bucket-pick" is STALE. CONTEXT.md D-89/D-90/D-91 explicitly defer the bank pick. The phase title is misleading.

**How to avoid:** Plan-phase MUST claim coverage of PIXLAB-01..04 only â€” NOT VAR-PIXEL-01. The closing SUMMARY documents VAR-PIXEL-01 as deferred (already in REQUIREMENTS.md v2 Requirements). No `variation_seed` field anywhere in the diff. No hash-pick code anywhere in the diff. Plan-checker MUST flag any task that adds a `variation_seed` property.

**Warning signs:** Plan tasks reference `variation_seed`. `mask_to_atlas` signature gains a `coord` parameter. `_first_cell_by_mask` becomes `_cells_by_mask: Array[Array]` (lists of cells per mask, the v2 shape).

### Pitfall 5: Forward-compat versioning machinery sneaking in

**What goes wrong:** Adding `@export var pixellab_version: int = 1` "in case PixelLab updates the tables."

**Why it happens:** Spike 003 README Â§"Edge cases" + PIXELLAB.md Â§"Constraints" both suggest this. CONTEXT.md D-92 explicitly REJECTS it. CLAUDE.md "Breaking Changes Policy (HARD RULE)" no-forward-compat clause bans it.

**How to avoid:** Plan-phase + plan-checker treat any `version: int` / `schema_version: int` / `format_version: int` field as a hard reject. CHANGELOG entries are the only acceptable compat work. The README "PixelLab interop" note (Phase 5 territory) can document version-coupling, but the runtime carries no version flag.

**Warning signs:** Diff adds `version` to a layout property. Diff adds `if pixellab_version < N:` branches. Diff adds a deprecation alias for an old `_CELL_TO_ROLE` table.

### Pitfall 6: Greybox PNG visible-gap regression

**What goes wrong:** PIXLAB greyboxes use partial-quadrant `draw_corner_mask` silhouettes (matches DualGrid16 convention). Single-grid rendering shows visible gaps at painted-region edges (Phase 2 UAT bug class #5).

**Why it happens:** Single-grid renders one tile per logic cell. Partial-fill silhouettes leave transparent quadrants between adjacent painted cells. The Phase 2 fix made `_paint_via_layout` skip non-logic-painted cells (line 272-273), preventing background extension â€” but the painted region's interior still shows the partial-fill silhouettes.

**How to avoid (option A â€” first try):** Use `draw_corner_mask` per role. The fix is silhouettes encoding the role's mask â€” cells with mask=15 (role 6) render solid 32Ă—32; cells with mask 0 (role 12) render fully transparent. If the visual regression test passes, ship it.

**How to avoid (option B â€” fallback):** Switch all PIXLAB greyboxes to solid 32Ă—32 (the Wang2Corner / Blob47Godot convention). Mask differentiation is by atlas POSITION only. Tradeoff: artist sees an opaque grid in inspector preview instead of role-coded silhouettes.

**Warning signs:** `pixellab_visual_regression_test.gd` reports "opaque-pixel bbox extends OUTSIDE expected user-painted bounds" or "transparent gaps inside painted region." `bitmask_bounds_test.gd` reports "pixels are TRANSPARENT inside the expected silhouette."

**Plan-phase resolution:** Plan-phase picks option A (mask-coded silhouettes â€” preserves PixelLab's role-shape encoding for inspector preview) and the visual regression test catches if it doesn't work. Fallback to option B is a 2-line generator change.

### Pitfall 7: Test fixture not committable due to Godot import pipeline

**What goes wrong:** The PixelLab sample PNGs are committed but `.import` files aren't generated. Test load fails at `load(_TOPDOWN_SAMPLE_PATH)` returning null.

**Why it happens:** Godot generates `.import` sidecar files on the first import pass (running with `--import` flag or opening the editor). Headless test runs don't trigger the import pipeline. Phase 3 noted this same issue: "Godot --import pass required for new bundled PNG to be load()-able at runtime" (STATE.md "Phase 3 Plan 04: Blob47Godot layout shipped" entry).

**How to avoid:** After copying the sample PNGs to `tests/pixellab_*_sample.png`, run `Godot --headless --path . --import` to generate `.import` files. Commit BOTH the PNG AND the `.import` file. Verify by running `pixellab_visual_regression_test.gd` headless â€” must load the sample without "could not load" failures.

**Warning signs:** Test failure: "could not load res://tests/pixellab_topdown_sample.png". Editor shows "Importing..." dialog on first open.
